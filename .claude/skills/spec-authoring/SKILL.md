# spec-authoring — 구현 명세 작성

기획 의도를 빌더·모션·햅틱·QA가 흔들림 없이 따라갈 단일 기준 문서로 번역한다. 정본은 `docs/PRODUCT_SPEC.md`.

## 왜 이렇게 쓰는가
이 앱은 한 제스처가 비주얼·사운드·햅틱 세 감각으로 동시에 반응한다. 명세가 그중 하나라도 빠뜨리면 담당 에이전트가 추측하게 되고, 감각이 어긋나 "촉각 환상"이 깨진다. 그래서 명세는 **감각 3종 세트 + 기기별 폴백**을 항상 함께 적는다.

## 작성 순서
1. **정본·현황 대조** — `docs/PRODUCT_SPEC.md`에서 해당 기능 섹션을 읽고, 기존 `lib/`에 재활용 가능한 코드가 있는지 확인한다(있으면 "재활용 후보"로 표시).
2. **범위 확정** — MVP 단계(스펙 7장: 필수 ①②③④ / 2차 ⑤⑥ / 확장 ⑦⑧⑨)에 비춰 포함·제외를 명확히. 요청을 넘는 기능은 추가하지 않는다.
3. **분해** — 화면(`Soothe/Pour/ReleaseSelect/ReleaseRitual/Closing`)·위젯·`RitualSession` 상태 변화로 쪼갠다.
4. **감각 3종 세트 작성** — 각 제스처/의식마다 비주얼·사운드·햅틱을 한 행으로. 햅틱은 스펙 5.1 카탈로그 패턴명으로 지정(`tapPop`, `shredGrind` 등).
5. **센서 매핑** — 모션 입력이 있으면 어떤 센서값(`userAccelerometer` 크기 등)이 무엇으로 변환되는지.
6. **폴백 명시** — iOS/Android/웹·미지원 기기별 graceful degradation.
7. **수용 기준** — QA가 그대로 체크할 검증 가능한 문장으로.

## 명세 템플릿
```markdown
# 명세: {기능명}
## 1. 정의 / 목적
## 2. 범위 (포함 / 제외 / MVP 단계)
## 3. 영향 파일 (신규 / 수정 / 재활용 후보)
## 4. 화면·위젯 동작 (상태·전환·RitualSession 필드)
## 5. 감각 3종 세트
| 제스처/의식 | 비주얼 | 사운드 | 햅틱(패턴명) |
|---|---|---|---|
## 6. 센서 매핑 (입력값 → 인터랙션)
## 7. RitualSession 데이터 흐름
## 8. 디자인 토큰 / 카피 요구
## 9. 기기별 폴백 (iOS / Android / 웹·미지원)
## 10. 수용 기준 (체크리스트)
## 11. 결정 필요 항목
```

## 원칙
- **제품 철학을 명세에 박는다:** "기다려주기"(자동 강요 금지, 제안만), 평가 카피 금지, 압박형 UI 금지. 위반 위험 요청은 차단 말고 위험·대안 병기.
- **추측 금지:** 모호하면 "결정 필요"로. 빈칸을 임의로 채우지 않는다.
- **검증 가능하게:** 수용 기준은 "부드럽다" 같은 모호어 대신 "tap 후 0.6s 이내 elastic 복원, light impact 1회" 식으로.
- 출력은 `_workspace/01_planner_{기능명}_spec.md`.

## 흔한 누락 (체크)
- 햅틱은 적었는데 사운드·폴백을 빠뜨림 → 3종 세트와 폴백은 한 묶음.
- `RitualSession`의 어느 필드가 언제 세팅/소비되는지 미기재 → QA 경계면 검증 불가.
- MVP 단계 미표기 → 확장 기능을 필수로 오해.
